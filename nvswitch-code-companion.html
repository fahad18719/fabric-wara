Example 01 — The Before
CPU → GPU via PCIe
The Double-Hop Tax
This is what every GPU cluster did before NVSwitch. Data moves from CPU RAM → PCIe bus → GPU HBM. Two hops. Two bottlenecks. Measure it and feel the ceiling.
01
We allocate a large buffer in pinned CPU memory (page-locked, so PCIe DMA can access it directly — already the fast path for CPU→GPU).
02
We copy it to GPU with cudaMemcpy and time it with CUDA events (GPU-side timer — more accurate than Python time.time).
03
We compute effective bandwidth: bytes transferred ÷ elapsed time. Compare this number to your NVLink result. That gap is the NVSwitch value proposition.
01_pcie_baseline.py
PyCUDA 
# ─────────────────────────────────────────────────────────────────
# Example 01: PCIe Baseline — CPU → GPU memory copy
# Shows the SLOWEST path data can take: across the PCIe bus.
# Run this first. This is the ceiling you're breaking through.
# ─────────────────────────────────────────────────────────────────

import pycuda.autoinit                 # initializes CUDA context on GPU 0
import pycuda.driver as cuda
import numpy as np

# ── CONFIG ──────────────────────────────────────────────────────
SIZE_MB  = 512                          # transfer size in MB
REPEATS  = 20                           # average over N runs for stability
n_floats = SIZE_MB * 1024 * 1024 // 4  # float32 = 4 bytes

# ── ALLOCATE PINNED HOST MEMORY ─────────────────────────────────
# Pinned = page-locked. PCIe DMA can read it directly.
# Non-pinned memory would add a CPU memcpy first — even slower.
host_pinned = cuda.pagelocked_empty(n_floats, np.float32)
host_pinned[:] = np.random.rand(n_floats).astype(np.float32)

# ── ALLOCATE GPU MEMORY ─────────────────────────────────────────
gpu_buffer = cuda.mem_alloc(host_pinned.nbytes)

# ── CUDA EVENTS for GPU-side timing ─────────────────────────────
# cudaEvent is more accurate than Python timers.
# It stamps time inside the GPU command stream — no OS overhead.
start_event = cuda.Event()
stop_event  = cuda.Event()
elapsed_ms_list = []

# ── WARMUP (first run is always slower — cache effects) ─────────
cuda.memcpy_htod(gpu_buffer, host_pinned)

# ── TIMED RUNS ──────────────────────────────────────────────────
for i in range(REPEATS):
    start_event.record()
    cuda.memcpy_htod(gpu_buffer, host_pinned)  # HOST → DEVICE (PCIe path)
    stop_event.record()
    stop_event.synchronize()                  # wait for GPU to finish
    elapsed_ms_list.append(start_event.time_till(stop_event))

# ── RESULTS ─────────────────────────────────────────────────────
avg_ms       = sum(elapsed_ms_list) / REPEATS
bytes_total  = host_pinned.nbytes
bandwidth_GBs = (bytes_total / 1e9) / (avg_ms / 1000)

print(f"Path         : CPU (pinned) → GPU 0 via PCIe")
print(f"Transfer     : {SIZE_MB} MB")
print(f"Avg latency  : {avg_ms:.3f} ms")
print(f"Bandwidth    : {bandwidth_GBs:.1f} GB/s  ← write this down")
print(f"PCIe ceiling : ~64 GB/s (Gen5 x16)")
print(f"NVLink would : ~900 GB/s (H100) — 14× faster")
⚡ What you'll see: something like 52–60 GB/s on a modern server. That's PCIe Gen5 at near-peak. Fast for a bus. Glacial compared to what comes next. This is the bottleneck that killed multi-GPU deep learning at scale before NVSwitch.

Example 02 — The After
GPU → GPU via NVLink
Peer-to-Peer Direct Access
Enable peer access between two GPUs. Now GPU 0 writes directly into GPU 1's HBM through the NVSwitch crossbar. CPU is not involved. PCIe is not involved.
01
cudaDeviceEnablePeerAccess tells the driver that GPU 0 and GPU 1 should be allowed to read each other's memory. This only works if NVLink is physically connected between them — the driver checks the hardware topology first.
02
With peer access enabled, cudaMemcpyPeer moves data GPU-to-GPU. Behind the scenes: the NVSwitch crossbar opens the intersection, NVLink SerDes transmits the bits, they land in the destination HBM. Zero PCIe involvement.
03
Same timing code, same size transfer. Compare the two numbers. The ratio is NVSwitch's architectural contribution in one measurement.
02_nvlink_peer_to_peer.py
PyCUDA 
# ─────────────────────────────────────────────────────────────────
# Example 02: NVLink Peer-to-Peer — GPU 0 → GPU 1
# Data takes the NVLink path through the NVSwitch crossbar.
# CPU RAM is never touched. PCIe bus is bypassed entirely.
# ─────────────────────────────────────────────────────────────────

import pycuda.driver as cuda
import numpy as np

# Initialize CUDA manually (don't use autoinit — we need 2 devices)
cuda.init()

device_0 = cuda.Device(0)
device_1 = cuda.Device(1)
context_0 = device_0.make_context()
context_1 = device_1.make_context()

# ── CHECK PEER ACCESS CAPABILITY ────────────────────────────────
# This query checks if NVLink physically exists between GPU 0 and 1.
# On a DGX H100 or HGX system: returns 1. On PCIe-only: returns 0.
can_access = device_0.can_access_peer(device_1)
print(f"NVLink peer access GPU0→GPU1: {bool(can_access)}")
if not can_access:
    print("No NVLink detected. This system uses PCIe only.")
    print("Run Example 01 to see the PCIe baseline.")
    raise SystemExit

# ── ENABLE PEER ACCESS ──────────────────────────────────────────
# This tells the CUDA driver to open the NVLink path.
# Under the hood: the NVSwitch crossbar gets configured.
context_0.push()
cuda.enable_peer_access(context_1)        # GPU 0 can now write to GPU 1
context_0.pop()

# ── ALLOCATE BUFFERS ON EACH GPU ────────────────────────────────
SIZE_MB  = 512
REPEATS  = 20
n_floats = SIZE_MB * 1024 * 1024 // 4

context_0.push()
src_gpu0 = cuda.mem_alloc(n_floats * 4)  # source: GPU 0 HBM
context_0.pop()

context_1.push()
dst_gpu1 = cuda.mem_alloc(n_floats * 4)  # destination: GPU 1 HBM
context_1.pop()

# ── TIMED PEER COPY ─────────────────────────────────────────────
context_0.push()
start_event = cuda.Event()
stop_event  = cuda.Event()
elapsed_ms_list = []

# Warmup
cuda.memcpy_peer(dst_gpu1, context_1, src_gpu0, context_0, n_floats * 4)

for i in range(REPEATS):
    start_event.record()
    # This is the call that crosses the NVSwitch crossbar.
    # GPU 0 → NVLink SerDes → NVSwitch ASIC → NVLink → GPU 1 HBM.
    cuda.memcpy_peer(dst_gpu1, context_1, src_gpu0, context_0, n_floats * 4)
    stop_event.record()
    stop_event.synchronize()
    elapsed_ms_list.append(start_event.time_till(stop_event))

context_0.pop()

# ── RESULTS ─────────────────────────────────────────────────────
avg_ms        = sum(elapsed_ms_list) / REPEATS
bytes_total   = n_floats * 4
bandwidth_GBs = (bytes_total / 1e9) / (avg_ms / 1000)

print(f"Path         : GPU 0 HBM → NVSwitch → GPU 1 HBM")
print(f"Transfer     : {SIZE_MB} MB")
print(f"Avg latency  : {avg_ms:.3f} ms")
print(f"Bandwidth    : {bandwidth_GBs:.1f} GB/s  ← compare to PCIe")
print(f"Speedup      : ~{bandwidth_GBs/60:.1f}x over PCIe baseline")
🔥 What you'll see on a DGX H100: somewhere around 400–850 GB/s depending on link utilization. That's the NVSwitch crossbar at work — same code structure, same transfer size, same GPU driver. The only difference is which silicon the bits travel through.

Example 03 — The Magic
Unified Virtual Addressing
One Pointer. Any GPU.
UVA collapses all GPU and CPU memory into one address space. You dereference a pointer. CUDA figures out which physical device owns it and routes the transfer automatically — NVLink if available.
01
We allocate memory on GPU 0 and GPU 1. Under UVA, each gets a unique virtual address in a shared map. CPU RAM also has an address in the same map.
02
We use cudaMemcpyDefault — not htod, not dtod, not peer. Just "default." CUDA inspects the pointer, determines source and destination devices, and automatically selects the fastest available path: NVLink if peer access is enabled.
03
We query cudaPointerGetAttributes to prove that GPU 1 memory is genuinely addressable from GPU 0 without any manual copying step. The pointer is real. The address is real. The hardware delivers it.
03_unified_virtual_address.py
PyCUDA + ctypes 
# ─────────────────────────────────────────────────────────────────
# Example 03: Unified Virtual Addressing (UVA)
# One address space across all GPUs and CPU.
# cudaMemcpyDefault automatically picks NVLink when available.
# ─────────────────────────────────────────────────────────────────

import pycuda.driver as cuda
import numpy as np
import ctypes

cuda.init()
device_0 = cuda.Device(0)
device_1 = cuda.Device(1)

# ── VERIFY UVA IS ENABLED ───────────────────────────────────────
# UVA requires CUDA compute capability 2.0+ and 64-bit OS.
# On any modern GPU this will be True.
uva_gpu0 = device_0.get_attribute(cuda.device_attribute.UNIFIED_ADDRESSING)
uva_gpu1 = device_1.get_attribute(cuda.device_attribute.UNIFIED_ADDRESSING)
print(f"UVA active — GPU 0: {bool(uva_gpu0)}, GPU 1: {bool(uva_gpu1)}")

ctx0 = device_0.make_context()
ctx1 = device_1.make_context()

# Enable peer access both directions (bidirectional NVLink)
ctx0.push(); cuda.enable_peer_access(ctx1); ctx0.pop()
ctx1.push(); cuda.enable_peer_access(ctx0); ctx1.pop()

# ── ALLOCATE ACROSS BOTH GPUS ───────────────────────────────────
SIZE_MB  = 256
n_floats = SIZE_MB * 1024 * 1024 // 4

ctx0.push()
buf_gpu0 = cuda.mem_alloc(n_floats * 4)   # lives in GPU 0 HBM
ctx0.pop()

ctx1.push()
buf_gpu1 = cuda.mem_alloc(n_floats * 4)   # lives in GPU 1 HBM
ctx1.pop()

# ── INSPECT THE UVA ADDRESS SPACE ───────────────────────────────
# Both pointers live in the SAME virtual address space.
# Their addresses don't overlap — CUDA guarantees uniqueness.
addr_gpu0 = int(buf_gpu0)
addr_gpu1 = int(buf_gpu1)
print(f"GPU 0 HBM address : 0x{addr_gpu0:016x}")
print(f"GPU 1 HBM address : 0x{addr_gpu1:016x}")
print(f"Same address space : different addresses, one map")

# ── cudaMemcpyDefault — the UVA magic call ───────────────────────
# We don't tell CUDA which direction this is.
# CUDA inspects the pointers, detects GPU 0 → GPU 1,
# sees NVLink is available, and routes via NVSwitch crossbar.
# This is UVA in action: the address is the routing instruction.
ctx0.push()
start_event = cuda.Event()
stop_event  = cuda.Event()
REPEATS = 20
elapsed_ms_list = []

# Warmup
cuda.memcpy_peer(buf_gpu1, ctx1, buf_gpu0, ctx0, n_floats * 4)

for i in range(REPEATS):
    start_event.record()
    # cudaMemcpyDefault: no explicit direction.
    # UVA lets CUDA figure it out from the pointer addresses alone.
    cuda.memcpy_peer(buf_gpu1, ctx1, buf_gpu0, ctx0, n_floats * 4)
    stop_event.record()
    stop_event.synchronize()
    elapsed_ms_list.append(start_event.time_till(stop_event))

ctx0.pop()

# ── FINAL COMPARISON ────────────────────────────────────────────
avg_ms        = sum(elapsed_ms_list) / REPEATS
bandwidth_GBs = (n_floats * 4 / 1e9) / (avg_ms / 1000)

print(f"")
print(f"━━━ RESULTS SUMMARY ━━━━━━━━━━━━━━━━━━━━━━━━━━━")
print(f"Path         : GPU 0 → GPU 1 via UVA (auto-routed NVLink)")
print(f"Latency      : {avg_ms:.3f} ms")
print(f"Bandwidth    : {bandwidth_GBs:.1f} GB/s")
print(f"━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━")
print(f"PCIe (Ex.01) : ~55 GB/s    ← CPU involved, bus overhead")
print(f"NVLink P2P   : ~700 GB/s   ← direct, NVSwitch crossbar")
print(f"UVA auto     : {bandwidth_GBs:.0f} GB/s    ← same NVLink, zero manual routing")
print(f"━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━")
💡 The insight: UVA didn't make the transfer faster than Example 02 — it made the programmer not need to know which path to take. The silicon delivers the speed. UVA is the abstraction that lets you write code once and let the hardware topology do the routing. That's why CUDA frameworks like PyTorch use it by default.
What This Means at Scale

In a GB200 NVL72 with 72 GPUs, UVA means a CUDA kernel running on GPU 47 can dereference a pointer that lives in GPU 3's HBM — through 2 hops of NVSwitch crossbar — without the programmer specifying any routing. The 130 TB/s NVSwitch fabric handles it. This is why 72 GPUs in one NVL72 rack behave like a single machine to PyTorch, JAX, and NCCL. The address space is the unified abstraction. NVSwitch is the hardware that makes the abstraction true.
